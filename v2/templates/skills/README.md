## project/

### fast/

- `init` — Быстрый старт планирования проекта: цель, границы и минимальные артефакты
- `pulse-scan` — Экспресс-скан контекста, рисков и внешних ограничений
- `stack-pick` — Быстрый и обоснованный выбор технологического стека
- `proto-spec` — Компактный спецификационный документ для одной итерации
- `task-blast` — Мгновенная декомпозиция proto-spec в исполнимые задачи
- `status` — Обязательное ведение прогресса fast-потока через `status.json`

### standart/

- `brief` — Формирование начального брифа для стандартного планирования
- `research` — Точечный ресерч для закрытия критичных неизвестных
- `proj-description` — Полный PRD для standard-flow (`prd.md`)
- `personas` — Описание и валидация персон для приоритизации требований
- `usecases` — Проработка use cases и пользовательских потоков с трассируемостью к PRD
- `arch` — Архитектурный дизайн по PRD/use cases и ключевым ограничениям
- `epic` — Декомпозиция в приоритизированные эпики с трассируемостью
- `decomposition` — Разбиение эпиков на задачи, готовые к реализации
- `status` — Обязательный трекинг стадий standard-flow через `status.json`

### discovery/

- `mode-selector` — Выбор режима планирования (`fast` или `standart`) по сложности, рискам и неизвестным

## planning/

- `approach-selection` — Выбор подхода к реализации (по рискам, срокам и сложности)
- `base` — Базовый каркас планирования задачи и структура итогового плана
- `change-inventory` — Инвентаризация изменений: что и где нужно менять
- `estimation` — Оценка объема и сроков работ
- `impact-analysis` — Анализ влияния изменений на модули, API и процессы
- `migration-strategy` — Стратегия миграций данных/схем и этапность перехода
- `monitoring-checks` — План проверок мониторинга и сигналов после релиза
- `requirements-extraction` — Извлечение и формализация требований из запроса
- `risk-assessment` — Выявление и приоритизация рисков с митигациями
- `rollback-mechanism` — Подход к откату изменений при проблемах
- `rollout-strategy` — Стратегия выката (этапы, аудитория, контроль)
- `scope-definition` — Определение границ задачи: in-scope / out-of-scope
- `scope-minimization` — Сужение объема до минимально ценного инкремента
- `task-classifier` — Классификация типа задачи для выбора правильного процесса
- `testing-strategy` — Определение тестовой стратегии для изменений

## testing/

- `a11y` — Проверки доступности интерфейсов (WCAG, навигация, семантика)
- `api-manual` — Ручное тестирование API через `curl` и воспроизводимые запросы
- `aqa` — Паттерны автоматизации тестирования (e2e/integration)
- `browser-manual` — Ручное UI‑тестирование в браузере и через DevTools
- `checklist` — Чеклисты для smoke/sanity/regression проверок
- `contract` — Контрактное тестирование API и совместимости схем
- `e2e-flow` — Сквозные пользовательские сценарии end-to-end
- `load` — Нагрузочное тестирование и базовые перф-бенчмарки
- `mobile` — Проверки мобильных сценариев, адаптивности и поведения
- `security` — Базовые проверки безопасности в тестовых сценариях
- `test-case` — Правила написания тест-кейсов (Given-When-Then)
- `visual-regression` — Визуальные регрессионные проверки интерфейса

## review/

### code/

- `strategy` — Стратегия code-review: фокус, глубина и порядок проверки
- `checklist` — Базовый чеклист code-review по качеству и корректности
- `requirements` — Проверка соответствия кода требованиям и критериям приемки
- `security` — Проверка кода на типовые уязвимости и риски безопасности
- `performance` — Проверка влияния изменений на производительность
- `maintain` — Проверка поддерживаемости: читаемость, связность, техдолг
- `idiom-check` — Проверка идиоматичности и соответствия стилю/практикам стека

### arch/

- `consistency` — Проверка архитектурной согласованности изменений
- `full-check` — Полный архитектурный ревью: границы, контракты, риски

### doc/

- `strategy` — Стратегия review документации: покрытие, ясность, актуальность
- `quality` — Проверка качества документации: структура, полнота, точность

## coder/

### frontend/

- `design` — Практики frontend design-системы: токены, типографика, цветовые роли, layout и консистентность компонентов
- `ui-ux` — Практики frontend UI/UX: пользовательские потоки, состояния интерфейса, доступность, формы и perceived performance
